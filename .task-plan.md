# Implementation Plan: g8t4

## Overview

**目标:** 用新的 `TaskRepo` 替代 `GoalTaskRepo`，统一 Task 概念。

g8t1 已完成数据库 Schema 迁移（`goal_tasks` → `tasks`，`goal_id` 改为 nullable），但代码层仍使用旧的 `GoalTaskRepo` + `IGoalTaskRepo` 接口（所有方法都需要 `goalId` 参数）。本任务需要：

1. **修复 `GoalRepo` 的表名 bug** — 仍引用已废弃的 `goal_tasks`/`goal_task_deps` 表名 + `thread_id` 列名
2. **创建 `TaskRepo`**（替代 `GoalTaskRepo`）— 新接口以 `taskId` 为主键，`goalId` 降为可选
3. **类型重命名** — `GoalTask` → `Task`，`IGoalTaskRepo` → `ITaskRepo` 等（保留过渡别名）
4. **Orchestrator 适配** — `OrchestratorDeps.goalTaskRepo` → `taskRepo`
5. **qdev 创建 task 入库** — 当前 qdev 只创建 Discord session 不写 tasks 表，改为写入 `goal_id=null` 的 task 行
6. **移除 `/task` 斜杠命令** — 功能已被 `POST /api/tasks` + qdev 覆盖
7. **更新 API 路由** — goal task 操作的 API 注释和参数名更新

## Key Decisions

### 1. 保留 deprecated 别名

`GoalTask`、`GoalTaskRepo`、`IGoalTaskRepo` 等保留为 deprecated 别名而非直接删除。原因：
- 避免一次性全量替换造成遗漏
- 渐进式迁移，后续 task 可以清理
- MCP 工具、skills 等外部消费者不会立即被破坏

### 2. TaskRepo.save() 的 goalId 参数策略

`save(task, goalId?)` 中 goalId 优先级：
1. 方法参数（如果非 undefined）
2. `task.goalId` 字段
3. 默认 null

这样 Orchestrator 可以继续传 goalId 参数（保持清晰），qdev 则不传或传 null。

### 3. qdev task ID 使用 channelId

与 Goal Drive 中 `task.threadId = channelId` 的模式一致。`findByChannelId()` 可以同时找到 goal 子任务和独立 qdev 任务。

### 4. GoalRepo 保持独立的 task 读写逻辑

`GoalRepo.save()` 内联写 tasks（在同一事务中），不委托给 `TaskRepo`。原因是 GoalDriveState 保存需要原子性（goal 元数据 + tasks 一起写）。GoalRepo 的 SQL 需更新表名和列名，但架构不变。

### 5. 移除 /task 但保留 POST /api/tasks

`/task` Discord slash command 与 `POST /api/tasks` API 功能完全重复。保留 API 路由（MCP 工具仍在使用），移除 Discord 命令。用户通过 `/qdev` 创建开发任务。

### 6. Task 接口新增 goalId 字段

运行时 `Task` 对象新增 `goalId?: string | null` 字段，由 `rowToTask()` 从 DB 读取。这使得 TaskRepo 不需要在每个方法中都传 goalId 参数——task 对象本身携带了关联信息。

## Edge Cases / Risks

### 1. GoalRepo 和 TaskRepo 对 tasks 表的并发写入

**风险:** `GoalRepo.save()` 直接写 tasks 表（inline SQL），`TaskRepo` 也写 tasks 表。同一 goal 上并发操作可能冲突。

**缓解:** Orchestrator 的 `withStateLock()` 已防止同一 goal 的并发状态操作。不同 goal 之间不会冲突（WHERE goal_id = ?）。风险低。

### 2. GoalRepo 读取 tasks 表的类型映射 bug

**风险:** migration 010 将 `thread_id` 列重命名为 `channel_id`，但 GoalRepo 的 SQL 和类型映射仍引用 `thread_id`。这是 **g8t1 遗留的现存 bug**。

**缓解:** Step 1 作为前置修复，在任何其他变更之前先修复此 bug。

### 3. TaskRepo.delete() 移除 goalId 条件的安全性

**风险:** 旧 SQL `DELETE FROM tasks WHERE goal_id = ? AND id = ?` 改为 `DELETE FROM tasks WHERE id = ?`。

**缓解:** migration 010 已将 id 改为全局唯一主键（单列 PK），不再需要 goalId 作为复合键的一部分。安全。

### 4. task_deps 表的 goal_id 可能为 null

**风险:** 独立 task（qdev）的 deps 中 goal_id 为 null。deps 查询 `WHERE goal_id = ?` 无法匹配 null。

**缓解:** 当前 qdev task 不会有 deps（`depends: []`），短期无风险。长期应改为 `WHERE task_id = ?` 而非 `WHERE goal_id = ? AND task_id = ?`。本次已在 Step 3 的 SQL 变更中处理。

### 5. /task 命令移除后的用户体验

**风险:** 用户习惯 `/task` 创建新任务。

**缓解:** `/qdev` 已覆盖创建开发任务的需求（更强大，自动创建 worktree + 触发 Claude）。`POST /api/tasks` API 仍可用。MCP 工具 `bot_list_tasks`、`bot_qdev` 不受影响。

### 6. 迁移后 GoalDriveState.tasks 中的 goalId 字段

**风险:** 现有 GoalDriveState 在内存中的 Task 对象没有 goalId 字段。添加后 GoalRepo 的 rowsToGoalDriveState() 需要填充它。

**缓解:** `rowsToGoalDriveState()` 中 task 映射已有 `goal_id` 来源（GoalRow.id），只需在映射中添加 `goalId: goal.id`。
