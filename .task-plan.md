# Implementation Plan: g8t1

## Overview

数据库 Schema 迁移，核心目标是**术语统一**——将当前混用的 "session/thread/channel" 概念拆分为清晰的实体：

1. **channels** — Discord Text Channel 实体（当前分散在 sessions.thread_id、goal_tasks.thread_id 中）
2. **claude_sessions** — Claude Code CLI 会话实体（当前嵌入在 sessions 表中）
3. **channel_session_links** — Channel 与 Claude Session 的多对多关联表
4. **sync_cursors** — 同步游标表（跟踪各数据源的同步进度）
5. **tasks** — 从 goal_tasks 重命名，goal_id 改为 nullable，使 task 成为独立实体

当前 `sessions` 表混合了三个概念：Discord Channel 信息（thread_id, guild_id）、Claude 会话状态（claude_session_id, plan_mode）、以及工作目录上下文（cwd, worktree_branch）。拆分后每个表职责单一，为后续 API/UI 层的术语统一打好基础。

## Current State Analysis

### 现有表关系

```
sessions (混合实体)
  ├── id (UUID)
  ├── thread_id (= Discord Channel ID)
  ├── guild_id
  ├── claude_session_id (= Claude CLI session)
  ├── cwd, worktree_branch (工作上下文)
  └── plan_mode, model (Claude 会话属性)

archived_sessions (sessions 的归档副本)
  └── 同 sessions + archived_at/by/reason

goal_tasks (隶属于 Goal)
  ├── PRIMARY KEY (goal_id, id)
  ├── thread_id (= Discord Channel ID)
  └── branch_name (git 分支)
```

### 目标表关系

```
channels (Discord Channel 实体)
  ├── id (TEXT PK = Discord Channel ID)
  ├── guild_id
  ├── name
  ├── cwd, worktree_branch
  ├── parent_channel_id
  └── status: 'active' | 'archived'

claude_sessions (Claude CLI 会话实体)
  ├── id (TEXT PK = UUID)
  ├── claude_session_id (Claude CLI session_id)
  ├── channel_id FK → channels.id
  ├── model, plan_mode
  └── status: 'active' | 'closed'

channel_session_links (关联表)
  ├── channel_id FK → channels.id
  ├── claude_session_id FK → claude_sessions.id
  └── linked_at, unlinked_at

tasks (原 goal_tasks，独立化)
  ├── id TEXT PK (全局唯一，如 "g2t1")
  ├── goal_id NULLABLE FK → goals.id
  ├── channel_id FK → channels.id (替代 thread_id)
  └── ...其余字段不变

sync_cursors (同步游标)
  ├── source TEXT PK
  └── cursor TEXT, updated_at
```

#### 1.1 创建 channels 表

```sql
CREATE TABLE channels (
  id                TEXT PRIMARY KEY,        -- Discord Channel ID (snowflake)
  guild_id          TEXT NOT NULL,
  name              TEXT NOT NULL DEFAULT '',
  cwd               TEXT NOT NULL,
  worktree_branch   TEXT,
  parent_channel_id TEXT,                    -- 父 Channel（fork 关系）
  status            TEXT NOT NULL DEFAULT 'active',  -- 'active' | 'archived'
  archived_at       INTEGER,
  archived_by       TEXT,
  archive_reason    TEXT,
  message_count     INTEGER NOT NULL DEFAULT 0,
  created_at        INTEGER NOT NULL,
  last_message      TEXT,
  last_message_at   INTEGER,
  UNIQUE(guild_id, id)
);

CREATE INDEX idx_channels_guild ON channels(guild_id);
CREATE INDEX idx_channels_status ON channels(status);
CREATE INDEX idx_channels_parent ON channels(parent_channel_id);
```

设计说明：
- PK 直接用 Discord Channel ID（之前 sessions 的 thread_id），不再需要额外 UUID
- 合并 sessions + archived_sessions 为一张表，通过 `status` 字段区分
- `archived_at/by/reason` 仅在 status='archived' 时有值
- `UNIQUE(guild_id, id)` 保持与现有查询兼容

#### 1.2 创建 claude_sessions 表

```sql
CREATE TABLE claude_sessions (
  id                      TEXT PRIMARY KEY,    -- 本地 UUID
  claude_session_id       TEXT,                -- Claude CLI session_id
  prev_claude_session_id  TEXT,                -- 上一轮（rewind 用）
  channel_id              TEXT REFERENCES channels(id),
  model                   TEXT,
  plan_mode               INTEGER NOT NULL DEFAULT 0,
  status                  TEXT NOT NULL DEFAULT 'active',  -- 'active' | 'closed'
  created_at              INTEGER NOT NULL,
  closed_at               INTEGER
);

CREATE INDEX idx_claude_sessions_channel ON claude_sessions(channel_id);
CREATE INDEX idx_claude_sessions_claude_id ON claude_sessions(claude_session_id);
CREATE INDEX idx_claude_sessions_status ON claude_sessions(status);
```

设计说明：
- `id` 沿用原 sessions.id（UUID）
- `claude_session_id` 是 Claude CLI 的真实 session ID
- 一个 Channel 可以有多个 Claude Session（clear 后创建新的）
- `channel_id` 外键关联到 channels 表

#### 1.3 创建 channel_session_links 表

```sql

  claude_session_id TEXT NOT NULL REFERENCES claude_sessions(id) ON DELETE CASCADE,
  linked_at         INTEGER NOT NULL,
  unlinked_at       INTEGER,

CREATE INDEX idx_csl_channel ON channel_session_links(channel_id);
CREATE INDEX idx_csl_session ON channel_session_links(claude_session_id);
```

设计说明：
- 记录 Channel 与 Claude Session 的关联历史
- 当前活跃链接：`unlinked_at IS NULL`
- 支持 attach 命令（将其他 session 关联到当前 channel）

#### 1.4 创建 sync_cursors 表

```sql
CREATE TABLE sync_cursors (
  source      TEXT PRIMARY KEY,     -- 数据源标识，如 'discord_messages', 'claude_sessions'
```

#### 1.5 重命名 goal_tasks → tasks，goal_id 改为 nullable

SQLite 不支持 RENAME TABLE 保留外键，需要重建：

```sql
-- 1. 创建新表
CREATE TABLE tasks (
  id              TEXT NOT NULL,
  goal_id         TEXT REFERENCES goals(id) ON DELETE CASCADE,  -- nullable
  description     TEXT NOT NULL,
  type            TEXT NOT NULL DEFAULT '代码',
  phase           INTEGER,
  status          TEXT NOT NULL DEFAULT 'pending',
  branch_name     TEXT,
  channel_id      TEXT REFERENCES channels(id),  -- 替代 thread_id
  dispatched_at   INTEGER,
  completed_at    INTEGER,
  error           TEXT,
  merged          INTEGER NOT NULL DEFAULT 0,
  notified_blocked INTEGER NOT NULL DEFAULT 0,
  feedback_json   TEXT,
  complexity      TEXT,
  pipeline_phase  TEXT,
  audit_retries   INTEGER NOT NULL DEFAULT 0,
  tokens_in       INTEGER,
  tokens_out      INTEGER,
  cache_read_in   INTEGER,
  cache_write_in  INTEGER,
  cost_usd        REAL,
  duration_ms     INTEGER,
  PRIMARY KEY (id)             -- 改为单列主键
);

CREATE INDEX idx_tasks_goal ON tasks(goal_id);
CREATE INDEX idx_tasks_status ON tasks(goal_id, status);
CREATE INDEX idx_tasks_channel ON tasks(channel_id);

-- 2. 迁移数据（将复合 ID 转为全局唯一 ID）
INSERT INTO tasks
  SELECT
    goal_id || ':' || id,       -- 组合为全局唯一 ID
    goal_id, description, type, phase, status,
    branch_name, thread_id,     -- thread_id 暂存到 channel_id
    dispatched_at, completed_at, error, merged, notified_blocked,
    feedback_json, complexity, pipeline_phase, audit_retries,
    tokens_in, tokens_out, cache_read_in, cache_write_in,
    cost_usd, duration_ms
  FROM goal_tasks;

-- 3. 重建 task_deps（注意 ID 映射）
CREATE TABLE task_deps (
  task_id             TEXT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  depends_on_task_id  TEXT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  goal_id             TEXT,        -- 保留 goal_id 方便查询
  PRIMARY KEY (task_id, depends_on_task_id)
);

INSERT INTO task_deps (task_id, depends_on_task_id, goal_id)
  SELECT
    goal_id || ':' || task_id,
    goal_id || ':' || depends_on_task_id,
    goal_id
  FROM goal_task_deps;
```

**重要决策 — task ID 映射**：

当前 goal_tasks 的 PK 是 `(goal_id, id)`，其中 `id` 如 "t1", "t2" 只在 goal 内唯一。改为单列 PK 后需要全局唯一。方案：

- 迁移时：`goal_id + ':' + id`（如 `"abc-uuid:t1"`）
- 新创建时：仍用 `gXtN` 格式（如 `"g2t1"`），因为 goal_seq 保证全局唯一
- 无 goal 的独立 task：使用 UUID

## Key Decisions

### 1. channels 表 PK 用 Discord Channel ID 而非 UUID

**理由**：Discord Channel ID 本身就是全局唯一的 snowflake，不需要额外的 UUID 层。直接用 Channel ID 做 PK 减少 JOIN 和映射开销。当前 sessions 表的 `id`（UUID）和 `thread_id`（Channel ID）之间的双 ID 设计是不必要的复杂性。

### 2. 合并 sessions + archived_sessions 为 channels 一张表

**理由**：当前 archive/restore 操作需要跨两张表移动数据（INSERT + DELETE），改用 status 字段切换后操作更简单，数据完整性更好。通过 `status` 字段 + 索引实现高效筛选。

### 3. goal_tasks → tasks 的 ID 映射策略

**理由**：当前 `(goal_id, id)` 复合主键中的 `id`（如 "t1"）只在 goal 内唯一。独立化后需要全局唯一 ID：
- 迁移数据：`goal_id + ':' + id`（确保不冲突）
- 新数据：继续用 `gXtN` 格式（如 `g2t1`），因为 goal.seq 已保证唯一
- 无 goal 的 task：使用 UUID

### 4. 废弃旧表而非立即删除

**理由**：重命名为 `_deprecated_*` 保留数据安全网。如果迁移出现问题可以回退。下一个 migration（011）可以真正 DROP。

### 5. task_deps 移除 goal_id 作为主键的一部分

**理由**：原 `goal_task_deps` 的主键是 `(goal_id, task_id, depends_on_task_id)`，因为 task_id 只在 goal 内唯一。现在 task_id 全局唯一后，主键简化为 `(task_id, depends_on_task_id)`。保留 `goal_id` 列方便批量查询，但不再是主键的一部分。

### 6. GoalTaskRepo 接口暂不改名

**理由**：本 task 只做 Schema 层的迁移。Repository 类名和方法签名的改名属于后续的「代码层术语统一」任务。本次保持 GoalTaskRepo 对外接口不变，只改底层 SQL。

### 7. channel_session_links 表的必要性

**理由**：虽然当前 claude_sessions 表已有 `channel_id` 外键，但 links 表记录了关联**历史**。这对于以下场景很重要：
- `/attach` 命令（将 session 从一个 channel 转移到另一个）
- 审计追踪（谁在什么时候使用了哪个 session）
- 未来支持多 session 并行切换

## Edge Cases / Risks

### 1. sessions.thread_id 重复

**风险**：如果 sessions 和 archived_sessions 中存在相同的 thread_id（一个活跃一个归档），插入 channels 表时会主键冲突。

**缓解**：使用 `INSERT OR IGNORE` 处理 archived_sessions 的迁移，活跃 session 优先。根据现有代码逻辑（archive 操作会先删 sessions 再插 archived_sessions），不应该出现重复。

### 2. goal_tasks ID 冲突

**风险**：不同 goal 下可能有相同的 task ID（如都有 "t1"），迁移到全局唯一 ID 时 `goal_id + ':' + id` 方案可能产生过长的 ID。

**缓解**：`goal_id` 是 UUID（36 字符）+ `:` + `tN`（2-3 字符），总长约 40 字符，TEXT 类型无限制。

### 3. interaction_log.session_id 关联断裂

**风险**：interaction_log 表的 `session_id` 存储的是 Claude CLI session_id，不是 sessions.id。迁移后这些引用仍然有效（指向 claude_sessions.claude_session_id）。

**缓解**：interaction_log 不需要修改——它引用的是 Claude CLI 的 session_id，与我们的表结构重组无关。

### 4. 运行时代码的兼容性

**风险**：StateManager、Orchestrator、API routes 大量使用 `threadId` 和 `sessions` 概念。Schema 改了但代码没改会崩溃。

**缓解**：GoalTaskRepo 内部改用新表名但对外接口不变（`threadId` 在运行时类型中保留，转换函数做映射）。Session 相关的代码层迁移是**后续任务**，不在本 task 范围。

**重要**：本 task 完成后，`SessionRepository` 仍然需要工作。有两个方案：
- **方案 A**：SessionRepository 内部改为查询新表（channels + claude_sessions），但对外返回 Session 类型
- **方案 B**：SessionRepository 暂时不改，继续查询旧表（_deprecated_sessions），等后续任务统一迁移

**推荐方案 A**——SessionRepository 改为新表的适配器，避免依赖已废弃的表。

### 5. SQLite 版本限制

**风险**：某些 ALTER TABLE 操作在旧版 SQLite 中不支持（如 RENAME TABLE 在 3.25+ 才支持）。

**缓解**：Node.js better-sqlite3 捆绑的 SQLite 版本通常是最新的（3.40+），支持 ALTER TABLE RENAME TO。在 migration `down()` 中使用 table rebuild 方式确保兼容。

### 6. 事务大小

**风险**：如果 sessions + archived_sessions 数据量很大，一个大事务可能导致 WAL 文件膨胀。

**缓解**：当前部署是单用户私有 Bot，数据量极小（通常 < 100 条 session）。不需要分批迁移。

## Migration Down() Strategy

`down()` 函数需要能够完整回滚：

1. 从新表恢复数据到旧表结构
2. 重命名 `_deprecated_*` 表回原名
3. DROP 新表

```sql
-- 恢复旧表名
ALTER TABLE _deprecated_sessions RENAME TO sessions;
ALTER TABLE _deprecated_archived_sessions RENAME TO archived_sessions;
ALTER TABLE _deprecated_goal_tasks RENAME TO goal_tasks;
ALTER TABLE _deprecated_goal_task_deps RENAME TO goal_task_deps;

-- 删除新表
DROP TABLE IF EXISTS channel_session_links;
DROP TABLE IF EXISTS claude_sessions;
DROP TABLE IF EXISTS channels;
DROP TABLE IF EXISTS task_deps;
DROP TABLE IF EXISTS tasks;
DROP TABLE IF EXISTS sync_cursors;
```
