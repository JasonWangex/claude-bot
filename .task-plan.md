# Implementation Plan: t3

## Overview

在 Claude 进程结束后，解析归档的 `.jsonl` 文件，提取每轮交互的结构化数据，写入 `interaction_log` 表。需要新建 JSONL 解析器模块 + InteractionLogRepository，并在 executor.ts 的进程退出流程中调用。

## JSONL 事件格式摘要

一个 JSONL 文件包含以下事件类型（按顺序出现）：

| 事件 | type | 特征 |
|---|---|---|
| 初始化 | `system` (subtype=init) | 含 `model`, `session_id`, `tools` |
| Assistant 文本 | `assistant` | `message.content[].type = 'text'`, 含 `message.usage` |
| Assistant 工具调用 | `assistant` | `message.content[].type = 'tool_use'`, 含 tool `name` |
| 用户/工具结果 | `user` | `message.content[].type = 'tool_result'` |
| 最终结果 | `result` (subtype=success) | 含 `total_cost_usd`, `modelUsage`, `num_turns`, `duration_ms` |

关键规则：
- **同一 `message.id`** 的多个 assistant 事件属于同一个 turn（如 text + tool_use 拆成两个事件）
- **`parent_tool_use_id` 非 null** 的事件属于子代理（subagent），应跳过（它们在子进程中有独立 JSONL）
- **`usage`** 位于 `message.usage`，仅 assistant 事件有
- **`model`** 位于 `message.model`

## Turn 定义

一个 "turn" 由以下组成：
1. 一个或多个 assistant 事件（相同 `message.id`）→ 产生一条 `role=assistant` 记录
2. 紧随的一个或多个 user 事件（tool_result）→ 产生一条 `role=user` 记录

特殊情况：
- 纯文本 assistant 回复没有后续 user 事件（最后一轮）
- `system:init` 和 `result` 事件不产生 interaction_log 记录（result 的聚合数据可在 session 级别使用）

## Files to Modify

- `discord/claude/jsonl-parser.ts` — **新建**，JSONL 解析器：读取文件、按 turn 分组、提取结构化数据
- `discord/db/interaction-log-repo.ts` — **新建**，InteractionLogRepository：批量写入 interaction_log 表
- `discord/db/index.ts` — 添加 InteractionLogRepository 的 re-export
- `discord/claude/executor.ts` — 在 `archiveOutputFiles` 后调用解析+入库逻辑
- `discord/types/db.ts` — 已有 `InteractionLogRow`，无需修改

## Implementation Steps

### Step 1: 新建 `discord/db/interaction-log-repo.ts`

遵循项目已有的 Repository 模式（参考 `devlog-repo.ts`）：

```typescript
import type Database from 'better-sqlite3';
import type { InteractionLogRow } from '../types/db.js';

export class InteractionLogRepository {
  private stmts: {
    insert: Database.Statement;
    findBySession: Database.Statement;
    deleteBySession: Database.Statement;
  };

  constructor(private db: Database.Database) {
    this.stmts = {
      insert: db.prepare(`
        INSERT OR IGNORE INTO interaction_log
          (session_id, turn_index, role, content_type, summary_text, model, tokens_input, tokens_output, cost_usd, jsonl_path, created_at)
        VALUES (@session_id, @turn_index, @role, @content_type, @summary_text, @model, @tokens_input, @tokens_output, @cost_usd, @jsonl_path, @created_at)
      `),
      findBySession: db.prepare('SELECT * FROM interaction_log WHERE session_id = ? ORDER BY turn_index, role'),
      deleteBySession: db.prepare('DELETE FROM interaction_log WHERE session_id = ?'),
    };
  }

  insertBatch(rows: Omit<InteractionLogRow, 'id'>[]): void {
    const tx = this.db.transaction((rows: Omit<InteractionLogRow, 'id'>[]) => {
      for (const row of rows) {
        this.stmts.insert.run(row);
      }
    });
    tx(rows);
  }

  findBySession(sessionId: string): InteractionLogRow[] { ... }
}
```

要点：
- 使用 `INSERT OR IGNORE` 保证幂等（UNIQUE 约束去重）
- `insertBatch` 用事务包裹提升性能
- 不需要 async — better-sqlite3 是同步的

### Step 3: 新建 `discord/claude/jsonl-parser.ts`

核心模块，负责：
1. 读取 JSONL 文件并逐行解析
2. 按 `message.id` 合并同一 assistant turn 的多个事件
3. 跳过 `parent_tool_use_id` 非 null 的子代理事件
4. 提取结构化数据生成 `InteractionLogRow[]`

#### 解析算法

```
输入: JSONL 文件路径, jsonl_path（相对路径，用于存库）
输出: InteractionLogRow[]

1. 读取文件，逐行 JSON.parse
2. 过滤掉:
   - type=system 的事件
   - type=result 的事件 (但提取 total_cost_usd 用于成本分摊)
   - parent_tool_use_id 非 null 的事件（子代理）
3. 按 message.id 分组 assistant 事件（相同 id 的事件合并为一个 turn）
4. 构造 turn 序列:
   - 遇到新 message.id 的 assistant 事件 → 新 turn 开始
   - 遇到 user 事件 → 关联到当前 turn
5. 为每个 turn 生成两条记录:
   - assistant 记录: 合并 content 摘要、累加 usage
   - user 记录: 汇总 tool_result 信息
```

#### 摘要生成规则（summary_text）

**assistant 记录**:
- 纯文本: 截取前 200 字符
- tool_use: 列出工具名 `"Read(file.ts), Edit(file.ts), Bash"`
- 混合: `"<截断文本> | Read(file.ts), Edit(file.ts)"`

**user 记录**:
- tool_result: 列出工具结果状态 `"tool_result x3 (1 error)"`
- 纯文本（用户输入，虽然 JSONL 中少见）: 截取前 200 字符

#### content_type 规则

- assistant: 只有 text → `'text'`; 只有 tool_use → `'tool_use'`; 两者都有 → `'text+tool_use'`
- user: `'tool_result'`

#### token / cost 计算

- `tokens_input` / `tokens_output`: 从 `message.usage.input_tokens` / `output_tokens` 累加（同 turn 多个事件去重，因为同一 message.id 的事件共享相同的 usage 值 — **只取一次**）
- `cost_usd`: 从 result 事件的 `total_cost_usd` 按 turn 的 output_tokens 比例分摊，或直接留 null（成本在 session 级别已有，per-turn 精确计算不现实）

**决定: cost_usd 字段在 assistant 记录中留 null，全局成本已在 ClaudeResponse.total_cost_usd 中。** 后续如有需求可补充。

### Step 4: 修改 `discord/claude/executor.ts`

在进程退出处理中，`archiveOutputFiles` 之后调用 JSONL 解析和入库逻辑。

需要修改的位置：

**4a.** 在 `executor.ts` 顶部导入:

```typescript
import { parseJsonlFile } from './jsonl-parser.js';

```

**4b.** 新增私有方法 `processInteractionLog`:
```typescript
private processInteractionLog(jsonlPath: string, sessionId: string): void {
  try {
    const db = getDb();
    const repo = new InteractionLogRepository(db);
    const relativePath = relative(join(thisDir, '../..'), jsonlPath);
    const rows = parseJsonlFile(jsonlPath, relativePath, sessionId);
    if (rows.length > 0) {
      repo.insertBatch(rows);
      logger.debug(`Saved ${rows.length} interaction_log rows for session ${sessionId}`);
    }
  } catch (e: any) {
    logger.warn(`Failed to process interaction log: ${e.message}`);
  }
}
```

**4c.** 在 `child.on('exit')` 回调中，`archiveOutputFiles` 调用后添加:

```typescript

const archivedJsonlPath = join(archiveDir, basename(outputFile));
this.processInteractionLog(archivedJsonlPath, lastSessionId);
```

注意：需要在 `archiveOutputFiles` 之后调用，因为文件已移动到 archive 目录。要让 `archiveOutputFiles` 返回归档后的路径，或直接在调用处拼接。

**4d.** `cleanupOutputFiles`（用于重连后的 orphan process）同样需要调用 `processInteractionLog`。在 `cleanupOutputFiles` 方法中，`archiveOutputFiles` 之后添加同样的逻辑。

## Key Decisions

1. **跳过子代理事件** — `parent_tool_use_id !== null` 的事件来自 Task 工具创建的子代理，它们有独立的 JSONL 文件。顶层 JSONL 只记录顶层交互。

2. **cost_usd 留 null** — 精确的 per-turn 成本需要对 Claude API 的 token 计价规则做复杂计算（cache read vs creation 价格不同），且 `result.total_cost_usd` 已覆盖 session 级别的成本统计。不值得在 t3 阶段做这个。

3. **INSERT OR IGNORE 保证幂等** — 如果同一个 JSONL 被意外重复处理（如 crash 重试），UNIQUE 约束会自动跳过已存在的记录。

4. **同步调用而非异步** — better-sqlite3 是同步的，且在进程退出回调中执行。解析 + 入库对 14-100 行的 JSONL 文件来说耗时微秒级。

5. **turn_index 从 0 开始** — 第一个 assistant 回复是 turn 0，与 `InteractionLogRow.turn_index` 注释一致。

6. **usage 去重** — 同一 `message.id` 的多个 assistant 事件共享相同的 `usage` 值（是消息级别的累积值，不是增量值）。只取第一个或最后一个事件的 usage 即可。根据实际观察，**每个事件的 usage 是该事件截止时的累积值**，所以取**最后一个**同 message.id 事件的 usage 最准确。

7. **UNIQUE 约束 `(session_id, turn_index, role)` 的兼容** — 一个 turn 只写两条记录：一条 assistant + 一条 user。不存在冲突。

## Edge Cases / Risks

1. **空 JSONL 或只有 system:init** — 进程被立即 kill。`parseJsonlFile` 返回空数组，不入库。无害。

2. **无 result 事件** — 进程超时或被 SIGKILL。解析正常进行，只是没有 cost 数据。已通过 cost_usd=null 处理。

3. **session_id 为空** — `archiveOutputFiles` 已处理此情况（skip archive）。如果 archive 未执行，`processInteractionLog` 也不会被调用。

4. **JSONL 解析错误（某行 JSON 损坏）** — 单行 try-catch，跳过坏行继续解析其他行。

5. **超大 JSONL（数千行）** — 理论上不会出现（Claude CLI 单次调用通常 <500 个事件），但如果出现，同步读取 + 解析仍然很快。不需要流式处理。

6. **子代理嵌套** — 子代理的事件混在顶层 JSONL 中，通过 `parent_tool_use_id` 过滤即可。子代理本身不产生独立的 JSONL 归档（它们没有独立的 executor 进程），所以**子代理的交互目前不会被记录**。这是可接受的 — 未来的 task 可以选择性记录子代理。

7. **DB 未初始化** — `getDb()` 在 bot 启动时已调用 `initDb()`。executor 是在 bot 运行时被调用的，DB 必然已初始化。但加 try-catch 保护。

8. **并发写入** — 同一时间可能有多个进程退出并写入 interaction_log。better-sqlite3 + WAL 模式 + busy_timeout 保证不会冲突。
