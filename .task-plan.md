# Implementation Plan: g8t3

## Overview

实现 Claude Session 同步服务，将 `~/.claude/projects/` 下的 `.jsonl` 会话文件**实时同步到 `claude_sessions` 表**。三条数据通路：

1. **实时写入**（execute 回调）：Bot 每次 Claude CLI 调用完成时，从返回的 `session_id` + JSONL 第一行元数据创建/更新 `claude_sessions` 记录
2. **定时扫描**（60s 间隔）：遍历 `~/.claude/projects/` 目录，用 `statSync(file).mtimeMs` 对比 `sync_cursors` 游标，发现新增/修改的 `.jsonl` 文件后提取元数据写入
3. **全量同步 API**：`POST /api/sync/sessions` 触发全目录扫描，适用于首次部署或数据修复

## 数据源分析

### Claude CLI 会话文件结构

```
~/.claude/projects/<encoded-cwd>/
├── <session-uuid>.jsonl           # 会话交互日志
├── <session-uuid>/                # 会话子目录（subagents、tool-results）
│   ├── subagents/
│   └── tool-results/
```

- **encoded-cwd**: 将工作目录路径的 `/` 替换为 `-`，加前缀 `-`。如 `/home/jason/projects/claude-bot` → `-home-jason-projects-claude-bot`
- **JSONL 第一行格式**（三种模式）：
  - `type: "queue-operation"` — 含 `sessionId`, `timestamp`（SDK 调用模式）
  - `type: "user"` — 含 `sessionId`, `cwd`, `version`, `gitBranch`, `timestamp`, `permissionMode`（直接 CLI 调用）
  - `type: "system", subtype: "local_command"` — 含 `sessionId`, `cwd`, `version`, `gitBranch`（CLI 本地命令）
- **第二行**（如果首行是 `queue-operation`）通常是 `type: "user"`，包含完整元数据

### 元数据提取策略

从 JSONL 文件的**前几行**（最多读 5 行）提取：
- `sessionId` — Claude CLI session UUID → `claude_sessions.claude_session_id`
- `cwd` — 工作目录 → 用于反查 `channels` 表的 channel
- `version` — Claude CLI 版本（可选，后续扩展用）
- `gitBranch` — git 分支名（可选）
- `timestamp` — 首条消息时间 → `claude_sessions.created_at`
- `message.model` — 从 assistant 事件提取模型名 → `claude_sessions.model`

## Implementation Steps

### Step 1: JSONL 元数据解析器 — `discord/sync/jsonl-metadata.ts`

创建独立的元数据提取模块，负责从 JSONL 文件首行读取会话元数据。

```typescript
interface SessionMetadata {
  sessionId: string;              // Claude CLI session UUID
  cwd?: string;                   // 工作目录
  model?: string;                 // Claude 模型
  gitBranch?: string;             // git 分支
  version?: string;               // Claude CLI 版本
  timestamp?: string;             // ISO-8601 时间
  permissionMode?: string;        // 权限模式
}

/**
 * 从 JSONL 文件提取会话元数据
 *
 * 读取策略：openSync + readSync 读取前 8KB → 按行解析 → 找到含 sessionId 的事件即停止
 * 不读取整个文件（可能很大），只取头部。
 */
export function extractSessionMetadata(jsonlPath: string): SessionMetadata | null
```

**实现要点**：
- 使用 `openSync` + `readSync` 读前 8KB（足够覆盖前 5 行），避免读取整个大文件
- 按 `\n` 分割，逐行 JSON.parse
- 优先从 `type: "user"` 事件提取（含最丰富元数据：`cwd`, `sessionId`, `gitBranch`, `version`）
- `type: "queue-operation"` 只有 `sessionId` 和 `timestamp`，作为 fallback
- 从 `type: "assistant"` 事件提取 `message.model`（如果前几行有的话）
- 返回 `null` 表示文件为空/损坏/不可读
- 错误处理：文件不存在、读取失败 → 返回 null，不抛异常

### Step 2: 同步服务主模块 — `discord/sync/session-sync-service.ts`

核心服务，管理三条同步通路。

```typescript
export class SessionSyncService {
  private scanTimer: NodeJS.Timeout | null = null;
  private claudeSessionRepo: ClaudeSessionRepository;
  private syncCursorRepo: SyncCursorRepository;
  private channelRepo: ChannelRepository;

  constructor(
    private db: Database.Database,
    private claudeProjectsDir: string,  // 默认 ~/.claude/projects
  )

  /** 启动定时扫描（60s 间隔） */
  start(): void

  /** 停止定时扫描 */
  stop(): void

  /** 实时同步单个会话（execute 回调调用） */
  syncSession(claudeSessionId: string, channelId?: string, model?: string): void

  /** 全量同步（API 调用），返回统计信息 */
  syncAll(): { discovered: number; created: number; updated: number }

  /** 定时扫描（内部调用） */
  private scanForChanges(): void

  /** 处理单个 JSONL 文件 */
  private processJsonlFile(jsonlPath: string): 'created' | 'updated' | 'skipped'
}
```

#### 2a. 实时同步 — `syncSession()`

当 `ClaudeExecutor.execute()` 完成并获得 `session_id` 后调用：

1. 用 `claudeSessionRepo.findByClaudeSessionId(claudeSessionId)` 查询是否已有记录
2. 如果不存在 → 创建新 `ClaudeSession`：
   - `id`: `randomUUID()`
   - `claudeSessionId`: 传入的 session_id
   - `channelId`: 传入的 channelId（对应 Discord Thread ID）
   - `model`: 传入的 model
   - `planMode`: false
   - `status`: 'active'
   - `createdAt`: Date.now()
3. 如果存在 → 更新 `model`、`channelId` 等字段（仅在值有变化时更新）
4. 调用 `claudeSessionRepo.save(session)`（upsert 语义）

**关键**：此方法是**同步调用**（SQLite 同步 API，repository 方法返回 Promise 但底层同步执行）。

#### 2b. 定时扫描 — `scanForChanges()`

每 60 秒执行一次，扫描 `~/.claude/projects/` 目录：

1. 从 `syncCursorRepo.get('claude_session_scan')` 读取游标
   - 游标值：上次扫描完成时的 Unix 时间戳（毫秒，字符串）
   - 首次运行时无游标 → 跳过扫描（等用户调 syncAll 或 execute 回调自然填充）
2. 用 Node.js `readdirSync` 列出 `claudeProjectsDir` 下的子目录
3. 对每个子目录，用 `readdirSync` + `statSync` 找出 `*.jsonl` 文件
4. 过滤 `mtimeMs > cursorMs` 的文件（仅处理有变动的）
5. 对每个文件调用 `processJsonlFile()`
6. 更新游标：`syncCursorRepo.set('claude_session_scan', String(Date.now()))`

**性能优化**：
- 外层 `readdirSync` 列目录名（无 stat 开销）
- 内层仅对 `.jsonl` 后缀文件调用 `statSync`（跳过子目录和其他文件）
- 使用 `try/catch` 包裹每个文件操作（单文件失败不中断整体扫描）

#### 2c. processJsonlFile — 处理单个文件

1. 调用 `extractSessionMetadata(jsonlPath)` 获取元数据
2. 如果 `sessionId` 为空 → 返回 `'skipped'`
3. 用 `claudeSessionRepo.findByClaudeSessionId(sessionId)` 查询
4. 不存在 → 创建新记录，返回 `'created'`
5. 存在 → 如有新信息（model 等）则更新，返回 `'updated'`
6. Channel 关联：用元数据中的 `cwd` 查询 `channels` 表（`channelRepo.getByGuild()` 遍历匹配 cwd）

#### 2d. 全量同步 — `syncAll()`

1. 遍历 `claudeProjectsDir` 下所有子目录 → 所有 `*.jsonl` 文件
2. 对每个文件调用 `processJsonlFile()`，统计 created/updated/skipped
3. 更新游标为当前时间
4. 返回 `{ discovered, created, updated }`

## Key Decisions

1. **不使用 chokidar/inotify，使用 readdirSync + statSync + mtime 游标**
   - 原因：`~/.claude/projects/` 下有 159+ 个子目录，inotify 监控开销大
   - 60s 定时扫描 + mtime 对比更可靠（不受 inotify watch limit 限制）
   - 纯 Node.js 实现，无 shell 依赖，跨平台兼容

2. **元数据仅读取 JSONL 前 8KB（非全文解析）**
   - 原因：JSONL 文件可能很大（数 MB），全文解析浪费资源
   - 会话 ID、CWD、模型等元数据全在前几行
   - 使用 `openSync` + `readSync` 定量读取

3. **实时同步优先，定时扫描兜底**
   - Bot 自身发起的 Claude 调用走 execute 回调（零延迟）
   - 手动执行 `claude` CLI（非通过 Bot）的会话走定时扫描（最大 60s 延迟）
   - 全量同步 API 用于修复/首次部署

4. **sync_cursors 游标策略**
   - source: `'claude_session_scan'`
   - cursor: Unix 毫秒时间戳字符串
   - 首次扫描时无游标 → 不做增量扫描（需显式调用 syncAll 初始化）

5. **channel 关联策略**
   - execute 回调：直接传入 `threadId`（= `channelId`），最可靠
   - 定时扫描：从 JSONL 元数据的 `cwd` 字段反查 `channels` 表
   - 找不到对应 channel → `channel_id = null`（独立会话，非 Bot 管理）

6. **onSessionSync 设计为可选回调属性而非构造参数**
   - `ClaudeExecutor` 不直接 import `SessionSyncService`（避免循环依赖）
   - 公共属性 + 回调函数注入，保持模块解耦
   - 回调缺失时（如 MCP Server 单独使用 executor）自动跳过

## Edge Cases / Risks

1. **JSONL 文件正在写入中**
   - 定时扫描可能读到不完整的首行
   - 缓解：`JSON.parse` 失败时 → 跳过该文件，下次扫描重试（mtime 会更新）

2. **相同 session_id 在多个项目目录出现**
   - Claude CLI 的 `--resume` 可能在不同 cwd 恢复同一 session
   - 处理：以 `claude_session_id` 为主键做 upsert，更新 `cwd`/`channel_id` 至最新值

3. **大量积压文件的首次全量同步**
   - 159 个项目目录 × 平均 10 个 JSONL 文件 ≈ 1590 次文件读取
   - 每次只读 8KB → 总 I/O 约 12MB，在同步 SQLite 操作中完全可接受
   - 如需更细粒度控制，可分批 `process.nextTick` 处理

4. **Bot 未运行时的手动 CLI 会话**
   - 仅存在于 `~/.claude/projects/` 中，不经过 Bot
   - 定时扫描和全量同步覆盖此场景
   - 这些会话的 `channel_id` 为 null（正常行为）

5. **executor detachAll 后的孤立进程**
   - Bot 重启后 `reconnectAll()` 在 `cleanupOutputFiles()` 中也触发 `onSessionSync`
   - 确保不丢失任何会话记录

6. **`claudeProjectsDir` 不存在**
   - 如果 `~/.claude/projects/` 不存在（首次安装 Claude CLI 前）
   - `readdirSync` 会抛异常 → `scanForChanges` 中 try/catch 捕获，不影响 Bot 运行
   - `start()` 正常注册 timer，等目录创建后自然开始工作

7. **定时扫描与全量同步并发**
   - 两者可能同时操作相同文件 → repository 的 upsert 语义保证幂等
   - 游标更新用 `syncCursorRepo.set()`（也是 upsert），无并发问题
