# Implementation Plan: g8t2

## Overview

Channel 管理服务的引入与 `threadId` → `channelId` 术语统一。分三个阶段：

1. **ChannelService** — 新建 Service 层，封装 Channel + ClaudeSession 的业务逻辑
2. **Discord 事件同步** — 监听 Discord Channel 事件，同步到 `channels` 表
3. **术语统一** — 全局将 `threadId` 重命名为 `channelId`，包括运行时类型、变量、函数名、API 响应字段

> g8t1 已完成数据库层（migration 010、ChannelRepository、ClaudeSessionRepository、SyncCursorRepository）。
> g8t2 聚焦于使这些新表在运行时生效，并统一术语。

## Implementation Steps

### Phase 1: ChannelService（核心 Service 层）

#### Step 1.1: 创建 ChannelService

新建 `discord/services/channel-service.ts`。

构造函数接收：
```typescript
constructor(
  private channelRepo: IChannelRepo,
  private claudeSessionRepo: IClaudeSessionRepo,
  private syncCursorRepo: ISyncCursorRepo,
)
```

核心方法：

1. **`ensureChannel(channelId, guildId, name, cwd, opts?)`** — 幂等 upsert Channel 记录
   - 如果 Channel 已存在且 status='archived'，自动 restore
   - opts 可传 parentChannelId、worktreeBranch
   - 返回 Channel 实体

2. **`archiveChannel(channelId, userId?, reason?)`** — 归档 Channel
   - 调用 channelRepo.archive()
   - 关闭关联的活跃 ClaudeSession
   - 返回 boolean

3. **`getOrCreateClaudeSession(channelId)`** — 获取或创建活跃 ClaudeSession
   - 查询 claudeSessionRepo.getActiveByChannel()
   - 不存在则创建新 ClaudeSession（UUID + channelId + status=active）
   - 返回 ClaudeSession

4. **`closeActiveSession(channelId)`** — 关闭 Channel 当前活跃 session

5. **`updateClaudeSessionId(localSessionId, claudeSessionId)`** — Claude CLI 返回后更新 claude_session_id

6. **`getActiveChannels(guildId)`** — 获取 Guild 下所有活跃 Channel

7. **`createVirtualChannel(guildId, name, cwd)`** — 创建后端虚拟通道
   - ID 格式：`virtual-${randomUUID()}`（与 Discord snowflake 明确区分）
   - 不创建 Discord Channel
   - 写入 channels 表

8. **`syncFromDiscord(discordChannel)`** — 从 Discord GuildChannel 对象同步到 DB
   - 提取 id, guild.id, name, parentId 等
   - upsert 到 channels 表
   - 返回 Channel 实体

#### Step 1.2: StateManager 双写改造

StateManager 构造函数新增可选参数：
```typescript
constructor(
  defaultWorkDir: string,
  private sessionRepo?: SessionRepository,          // 旧（保留兼容）
  private guildRepo?: GuildRepository,
  private channelRepo?: ChannelRepository,           // 新
  private claudeSessionRepo?: ClaudeSessionRepository, // 新
)
```

改造重点：
- **`load()`**: 如果新 Repo 存在，**同时**从 channels + claude_sessions 加载。如果新表有数据则优先使用，否则 fallback 到旧表
- **`persistSession()`**: 如果新 Repo 存在，**额外**写入 channels + claude_sessions 表（双写）
- **`getOrCreateSession()`**: 同步在 channels 表创建记录
- **`archiveSession()`**: 同步更新 channels 表 status='archived' + 关闭 ClaudeSession
- **`setSessionClaudeId()`**: 同步更新 claude_sessions 表

### Phase 2: Discord 事件同步

#### Step 2.1: 注册事件监听

在 `discord/bot/discord.ts` 的 `registerHandlers()` 中新增：

```typescript
this.client.on(Events.ChannelCreate, async (channel) => {
  if (!channel.guild || !checkAuth(channel.guild.id)) return;
  if (channel.type !== ChannelType.GuildText) return;
  await this.channelService.syncFromDiscord(channel);
});

this.client.on(Events.ChannelUpdate, async (oldChannel, newChannel) => {
  if (!newChannel.guild || !checkAuth(newChannel.guild.id)) return;
  if (newChannel.type !== ChannelType.GuildText) return;
  await this.channelService.syncFromDiscord(newChannel);
});

this.client.on(Events.ChannelDelete, async (channel) => {
  if (!channel.guild || !checkAuth(channel.guild.id)) return;
  await this.channelService.archiveChannel(channel.id, undefined, 'Discord channel deleted');
});
```

#### Step 2.2: 启动时全量同步

在 `launch()` 方法中，Bot ready 后对 authorized guild 做全量同步：

```typescript
const guild = await this.client.guilds.fetch(guildId);
await guild.channels.fetch(); // 填充缓存
const textChannels = guild.channels.cache.filter(c => c.type === ChannelType.GuildText);
for (const [, channel] of textChannels) {
  await this.channelService.syncFromDiscord(channel);
}
await this.syncCursorRepo.set('discord_channels', String(Date.now()));
```

> 全量同步只做 upsert，不删除 DB 中已有但 Discord 不存在的 Channel（可能是虚拟通道）。

## Key Decisions

1. **ChannelService 补充而非替代 StateManager** — StateManager 保持内存缓存 + 事件驱动架构，ChannelService 提供 Channel 实体的持久化层。StateManager 的 Session 概念逐步迁移到 Channel + ClaudeSession 双实体模型。ChannelService 不管理内存状态和锁，不处理消息队列——这些仍是 StateManager + MessageHandler 的职责。

2. **双写过渡期** — Phase 1 期间 StateManager 同时写入旧表和新表。新表为权威数据源。确保回退能力——如果新代码有 bug，旧表数据仍然完整。Phase 3 完成后可以在后续 task 中停止旧表写入。

3. **术语统一一步到位** — 不做渐进式 alias，直接全局替换。理由：
   - 代码库规模可控（单人项目）
   - TypeScript 编译器会帮助发现所有遗漏
   - 渐进式会引入 threadId/channelId 混用的混乱期

4. **虚拟通道 ID 用 `virtual-` 前缀** — 与 Discord snowflake（纯数字 18-20 位）明确区分。代码中可通过 `channelId.startsWith('virtual-')` 判断。所有 Discord API 调用前应检查非虚拟通道。

5. **API 路由参数同步更名** — `:threadId` → `:channelId`。内部 API，破坏性变更可接受。Web 前端、MCP 工具同步更新。

6. **Discord 事件同步采用最终一致性** — 实时事件监听 + 启动时全量同步。sync_cursors 仅记录最后同步时间用于审计/调试，不做复杂增量同步。

7. **Phase 3 和 Phase 1/2 在同一 task 中完成** — 因为术语统一与 ChannelService 紧密相关（例如 ChannelService 的方法都用 channelId，而当前代码传入的是 threadId），分开做会导致不一致。

## Edge Cases / Risks

1. **大规模重命名遗漏** — ~40 个文件涉及 threadId。**缓解**：TypeScript 编译器在类型层变更后报出所有引用点；Phase 3.13 的全局搜索二次验证。

2. **StateManager 双写一致性** — 旧表和新表数据可能不一致。**缓解**：新表为权威数据源，旧表仅用于兼容读取。写操作以新表为准，旧表写失败不阻塞主流程。

3. **虚拟通道 vs Discord API** — 虚拟 Channel 的 ID 不应传给 Discord API。**缓解**：所有 Discord API 调用（如 `client.channels.fetch(channelId)`、`channel.send()`、`mq.send(channelId, ...)`）前检查 `!channelId.startsWith('virtual-')`。当前 mq（MessageQueue）直接用 channelId 调 Discord，虚拟通道需要特殊处理（Phase 1 暂不强制引入虚拟通道到 orchestrator）。

4. **进程注册表兼容** — ProcessRegistryEntry.threadId 改名后，如果 Bot 重启时有旧格式的注册表文件，重连逻辑会找不到 threadId 字段。**缓解**：executor.ts 的重连读取逻辑加 fallback：`entry.channelId || (entry as any).threadId`。

5. **Web 前端与后端同步** — API 响应字段名 `thread_id` → `channel_id` 是破坏性变更。**缓解**：前后端在同一 commit 同时更新。部署时同时重启。

6. **GoalDriveState 持久化的 goalThreadId** — goals 表中序列化的 GoalDriveState JSON 包含 `goalThreadId` 字段名。GoalRepo 的 row 映射需要同步更新。已有数据的 JSON 中仍是 `goalThreadId`，反序列化时需要 fallback。**缓解**：GoalRepo 的 `rowToState()` 函数加 `goalChannelId: json.goalChannelId || json.goalThreadId` fallback。

7. **GoalTask 持久化的 threadId** — tasks 表 DB 列已是 `channel_id`（g8t1 完成），GoalTaskRepo 的映射已处理。运行时类型 `GoalTask.threadId` 改为 `channelId` 后，GoalTaskRepo 的 row 映射直接改为 `channelId: row.channel_id`。

8. **Orchestrator 的 replanner.ts** — 需要检查 replanner.ts 是否有 threadId 引用。如有则一并更新。
